CSS is hard to refactor


Not like with programming languages, where you can refactor at any time
Refactoring legacy Java project is easy, you have tests
For a huge legacy CSS, the best thing you can do is to rewrite it
There is no way to refactor this efficiently, especially if it's just a bunch of spaghetti rules without any general conception in mind
And rewriting CSS from scratch is not something that any manager will let you do (unless redesigning the website)

How can we solve this problem?

We can write tests for CSS

The first idea would be to find a way to write tests for CSS
Since tests for programming languages will greatly help you refactoring it, same should work for CSS
I don't mean tests that check if a given element has a class or a specific rule
No, I mean actually testing how the website looks
You modify CSS, open the browser, log in, click some buttons, resize for mobile and tablet

Find 5 differences...

It's like playing this game when you have 2 pictures that look almost the same and your task is to find 5 differences

...100 times

Except that you would be playing the same game over and over again with different kinds of changes
Tedious and boring, if you have more than few pages you will miss some bugs only to find them later with no idea which change broke them
Since human are not really good at this game, let's have machines play it
That's where the visual regression testing comes in

Visual regression test


Takes a screenshot
You make a modification, run test and make another screenshot
Compare pixel by pixel
Show diff

Challenges of visual regression testing:



  • Dynamic content
    (changing number or random blog post)

  • Sharing tests with your team
    (different OS, different graphic card, different configuration)
Visual regressions testing sounds great but there are problems with it
On top of the normal issues with tests which is that you have to update them over and over again, visual regressions have some other culprits
If you have a random element on you website a number or random blog post?
Well, then probably your tests will fail each time that number or that blog post changes
You can select what to compare
If not a part of CI, you have to share screenshots with team
Different OS and graphic card? Fail
Different configuration (Chromium instead of Chrome)? Fail
CI solves those issues
That's how we protect ourselves

Functional CSS is:



  • Pure (no side effects)
  • Composable
  • Immutable
  • Transparent (rem vs em)
Pure - class for background color sets the background color only
Pure - in standard CSS one class changes 10 properties and 5 in children due to cascading
Composable - small, reusable blocks
Immutable - property once declared, won't change (no specificity fight)
Transparent - you know what a class does, no matter where on page
