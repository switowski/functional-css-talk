CSS is hard to refactor


Not like with programming languages, where you can refactor at any time

How can we solve this problem?

We can write tests for CSS

The first idea would be to find a way to write tests for CSS
Since tests for programming languages will help you when you modify the code, same should work for CSS
I don't mean tests that check if a given element has a class or a specific rule
No, I mean actually testing how the website looks
You modify CSS, open the browser, log in, click some buttons, resize for mobile and tablet

Find 5 differences...

It's like playing this game when you have 2 pictures that look almost the same and your task is to find 5 differences

...100 times

Except that you would be playing the same game over and over again with different kinds of changes
Tedious and boring, if you have more than few pages you will miss some bugs only to find them later with no idea which change broke them
Since human are not really good at this game, let's have machines play it
That's where the visual regression testing comes in

Visual regression test


Takes a screenshot
You make a modification, run test and make another screenshot
Then it compares those 2 screenshots, pixel by pixel and if something is different, the test is failing, usually presenting you with a diff between these two images

PhantomCSS

There are many tools for visual regression tests, starting from the most well known - PhantomCSS
Install locally
Write some tests where you basically define the setup part, like log in, take screenshot
Take baseline screenshot
If there are some changes between the two screenshots, you will get a failing test
Compare and replace the baseline image or fix your CSS

spectre

Writing tests only for the visual regressions can be annoying
You already have to write unit, functional, integration, acceptance and god only know what kind of tests
For some programming languages you can find tools that will allow you to integrate visual regression testing with you existing functional tests
You are using the cucumber testing framework, there is specter
It will give you a nice web interface and you can send the screenshots directly from the cucumber functional tests

percy.io

But what is a real kicker for me is a way to add visual regression testing as a part of your continuous integration cycle and that's possible with percy.io
You can easily integrate percy.io with Travis, Jenkins or other popular CI tools
It supports Python, Ruby, and Ruby on Rails, Ember and static websites
Features like freezing animations
I have tried them before and there were some annoying bugs that they seems to fix recently, so I hope it's quite usable right now

How can we actually write the CSS, so that it's easier to maintain in the future?
Bunch of methodologies were introduced to help you with that.

Object Oriented CSS


1. Separate style from structure


.button {
    width: ...;
    height: ...;
    overflow: ...;
}

...

.alert {
    border: ...;
    background: ...;
    color: ...;
}
You should create separate classes for the structure that will define how the button or a modal looks like and separate classes for the skin of those elements
Like an alert button will have a red color and info button will have blue color
If you use frameworks like Bootstrap, you should be familiar with this pattern as this is exactly what Bootstrap is doing

Object Oriented CSS


2. Separate content from container


// BAD
.sidebar h3 {
    font-family: ...;
    font-size: ...;
}

// GOOD
.fancy-header {
    font-family: ...;
    font-size: ...;
}
Second rule - separate content from container - means that an element should look the same, no matter where it's located in the website
Don't style tags, style classes

Block, Element, Modifier


Think of HTML and CSS in 3 categories: block, element, modifier
Any independent element of your website is a block, like the menu block
All the pieces of code that live inside that block and don't make much sense outside of it are Elements, for example menu elements that live inside the menu block
Apply some modifications to the block or the element (active menu item), you use the modifier
Very simple, but after using it for a while, I can say that it's super efficient
Greatly improves encapsulation and reusability

Scalable and Modular Architecture for CSS (SMACSS)


  • Base
  • Layout
  • Module
  • State
  • Theme
More fine-grained categorization of elements
Encourages you to split the CSS classes into 5 different categories
Although I have no experience using this one, so I can't say how it works in practice

Functional CSS is:



  • Pure (no side effects)
  • Composable
  • Immutable
  • Transparent (rem vs em)
Pure - you can be sure that a class that is supposed to set the background color just sets a background color
Compare it with the standard CSS, where each class will change 10 properties and 5 more changes will be inherited from the parent
Composable - just like in a functional programming, you are building your element from small, reusable blocks
Immutable - you are 100% sure that once you declare a property, nothing will overwrite it (no specificity fight)
Transparent - you know exactly what will happen if you apply a given class
No matter where you put a given element on a page, if you add class with 1 rem font size, it will have this exact font size

Frameworks



  • Tachyons
  • Basscss
Don't write framework yourself
Two main, existing frameworks: Tachyons and Basscss, are pretty great
They differ slightly in the way they name classes but both are nice and have a very good documentation
Documentation is very important when using a functional CSS (otherwise you will start duplicating CSS)

The good parts


  • Modifying existing styling is easier
  • No more wrestling with the specificity
  • Very easy to add themes
  • Coding is faster
  • Encourages HTML components
    (Tachyons components )
  • Encourages design consistency
  • Cache-friendly
  • Faster websites
You can modify styles for HTML elements without the fear that you will break something (image that no longer needs to float to the right example)
If you remove a class or a CSS rule, you need to check other places if everything still works fine (cascading, so there can be consequences that you won't easily see), so developers will leave the old CSS code
You no longer have to fight with the specificity when you add a new rule just because Bootstrap had to chain 5 classes together to make something work
All classes have the same specificity level (which is just one class) - it will make modifications way easier. Adding RTL support is easy (or different theme colors)
Good luck writing a template for Bootstrap
Once you learn the most common class names, you will be actually coding way faster (div with small margin on top and black background example)
If you have to write the same set of classes over and over again, you will quickly get bored and you will probably create an HTML component (and that's good, it makes your code DRY)
Before you start working on a project, you go through the clean framework and adjust all settings according to the design vision (different heading sizes according to some typeset tools, make the different margin sizes consistent)
Then, you will start coding and you no longer will have to think how many pixels you need for margins
You are no longer distracted with the small design decisions when coding.
When you change the styling by changing the CSS classes in the HTML, you won't have problems with invalidation the cache
You might actually find out that after switching to functional CSS, you not only have less CSS files to download, but the overall size of your website went down, since you no longer duplicate the CSS rules

    When you switch to functional CSS, instead of duplicating the CSS rules, you will duplicate the CSS classes
    So instead of having a messy CSS files, we now have a messy HTML files, right
    You can and you should create HTML components for the parts of your website that are repeated often
    Every framework provides you with some way to create components

    The bad parts



    • "It's just replacing duplicated CSS rules with duplicated CSS classes!"
    • Updating existing components can be a pain
    • Won't work with non-functional frameworks
    Imagine that your designer decides that we should change the font family of multiple elements and maybe add more margin to all buttons
    With standard way of writing the CSS, if you separate the classes using some of the existing methodologies, you can just open the CSS files concerned with your changes and use find-replace to quickly apply your changes
    With functional CSS, you will have to modify all the HTML components and replace the existing classes, which indeed can take more time
    And probably the biggest problem for many developers is that it won't work with Bootstrap or other non-functional CSS that give you a lot of styling out of the box
    Bootstrap relies on cascading, functional CSS doesn't
    Bootstrap rules have higher specificity
    You can use !important

.modal {
    position: fixed;
    top: 50%;
    left: 50%;
    -webkit-transform: translate(-50%,-50%);
    -ms-transform: translate(-50%,-50%);
    transform: translate(-50%,-50%);
    *width: 600px;
    *margin-left: -300px;
    *top: 50px;
}
    And Functional CSS is not a religion, it's a tool
    In my opinion, if you really want to have CSS classes with a set of rules instead of HTML components, nothing stops you
    You can create a CSS for a generic element that you know it won't change often and you want to reuse in multiple places
    For example, if you want a modal for your website, just create this modal class
    Then if you need a bigger modal, just add 'width-50' class
    If you put the CSS in a correct order and you don't increase the selectors specificity, you shouldn't get into many problems
    To actually make it better, just add the styles that you know you won't need to overwrite and then add the remaining styles with functional classes

    Functional CSS is not a silver bullet

    Functional CSS is not a silver bullet (there are no silver bullets in CSS, there are just less painful ways of doing stuff), but it might solve some of the problems that you have.
